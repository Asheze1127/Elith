# 設計メモ：Elith RAG（企業向けRAGチャット基盤）

ELITH LOCK: FDE Challenge 2026-06 の課題に対する設計メモ。
標準RAGチャットを基盤に、3社の顧客要望から **A社を選んで実装**し、他社にも再利用すべき部分を共通機能として戻す。

---

## 1. 何を解決したい？

### 困っていること（いま何が問題？）

- 顧客（A/B/C社）はそれぞれ要望が異なるが、**顧客ごとに別プロダクト（フォーク）を作ると分岐が増えて破綻する**。共通修正を全社に配れず、改善が顧客に閉じ込められる。
- 一方で、要望をそのまま受けて作る受託開発では、再利用できる共通資産が育たない。
- RAGとして「根拠に基づく回答」「根拠不足時に断定しない」「古い/矛盾する資料の扱い」をきちんと構造化できていないと、回答品質が担当者依存になる。

### だれが困る？

- 顧客の問い合わせ担当者（回答の粒度・根拠がばらつく）
- プロダクト/FDE側（顧客が増えるたびにコードが増える）

### うまくいったと言える条件（できれば数字）

- **新規顧客の追加が「コード変更ゼロ（設定行の追加のみ）」で成立する**ケースが大半を占める
- A社要望が標準RAG + 設定で表現でき、ローカルでサンプルデータから動作確認できる
- 「顧客固有 / 設定で吸収 / 共通へ戻す」の切り分けがコードと設定に反映され、判断メモで説明できる

---

## 2. スコープ定義（やらないこと / Nice to have）

### 今回やること

- 標準RAGチャットの最小機能（取り込み・検索・引用付き回答・根拠不足の非断定・チャット画面・境界設計・ローカル動作）
- **A社向け追加機能**（引用必須 / 根拠不足ステータス / 古資料警告 / 資料矛盾表示 / フィードバック / 改善候補一覧 / カテゴリ別方針 / 社外社内モード）
- 残り2社も踏まえた **共通化判断**（標準RAGへ戻す/設定で吸収/顧客固有で残すの仕分け）

### 今回やらないこと

- B社・C社の実装（要望は構造化のみ。設定で表現できることを示す）
- リポジトリのフォーク、テナントごとのコード分岐
- 本番GCPへの実デプロイ（READMEに想定構成を記載するに留める）

### Nice to have（余裕があれば対応）

- B社：部署/ライン/設備ごとの検索スコープ、安全注意事項の差し込み
- C社：医療判断への回答制限、個人情報入力への注意、拠点別運用差分
- 継続的な回答品質改善ループ（フィードバック→ナレッジ更新）

---

## 3. アイデア

### こうする：

- **config駆動マルチテナント**。`/chat` API と RAGパイプラインは全テナント共通の1本。
- 顧客差分は `Depends` が注入する **`tenant_config`（DBの1行 = データ）** として吸収する。
- パイプラインの「部品（step）」は共通カタログ。**どの部品を・どの順で・どのパラメータで** 使うかを config が選ぶ。
- 詳細・スキーマは `details/multi-tenant-design.md`。

### なぜそうする？

- FDEとして評価されるのは機能数でなく **「顧客固有・設定吸収・共通還元」の切り分け** だから。
- 差分をデータに寄せると、新規顧客は設定行の追加で済み、共通改善が全社に届く。

---

## 4. 導入によって生じる変化

### 開発フローの変化：

- 「顧客が来た→フォーク/分岐を作る」から「顧客が来た→設定行を書く。足りない能力だけ共通部品を1個足す」へ。

### プロダクトの変化：

- 能力のカタログ（共通部品）が育ち、新規顧客対応のコストが逓減する。

### 担当者個人の変化（心理的側面）：

- 回答に根拠・更新日・警告が常に付くため、担当者が「これを根拠にしてよいか」で迷わなくなる。

---

## 5. 作業の進め方

### システム構成イメージ

- Frontend: Next.js（薄いチャットUI、config駆動レンダリング）
- Backend: FastAPI（`Depends` → 共通パイプライン → プロンプト層）
- Store: PostgreSQL + pgvector（リレーショナル + ベクトルを1つに）
- LLM/Embedding: Provider抽象（APIキー無し環境向けに mock 実装）

### 開発ステップ（MVPアプローチ）

1. 標準RAGを設計・実装（取り込み→検索→引用付き回答→根拠不足の非断定）
2. 3社要望を構造化（`feature-list.md`）
3. A社を選び、`tenant_config` + 必要な共通部品を実装
4. 残り2社も踏まえ共通化判断、標準RAG側へ反映
5. README（セットアップ / 設計判断 / GCP想定構成）と判断メモを整備

---

## 6. 分からないこと・懸念事項

- 「根拠不足の判定」「資料矛盾の検出」は config フラグでは済まず、共通パイプライン側に実ロジックが要る（精度の作り込みが論点）
- 取り込み（チャンク分割・埋め込み・索引化）を同期/非同期どちらで持つか

> 解決済み：**テナント解決はセッション（ログイン）で確定**。認証前にテナントを知る要件（別SSO・事前テーマ）が無く、デプロイもしないため。サブドメインは将来 独自URL/別IdPが要る場合の選択肢として残すのみ。
