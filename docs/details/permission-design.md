# 🔐 権限・境界設計

「誰が・どのデータに到達でき・どんな回答を出してよいか」を、テナント境界と属性ベースのポリシー（`tenant_config`）で制御する。

---

## 0. 設計前提

| 項目 | 内容 |
|---|---|
| 分離の単位 | `tenant_id`。全クエリをスコープする |
| ポリシーの所在 | `tenant_config`（属性ベース）。コードに顧客名を埋めない |
| 出力制御 | 回答モード（社外/社内 等）で情報の出し分け |

---

## 1. 用語定義

| 用語 | 意味 |
|---|---|
| テナント | 顧客（A/B/C社）。データとconfigの最上位境界 |
| ワークスペース | テナント内の区画（部署/拠点/ライン等） |
| 回答モード | 社外向け / 社内確認用 などの出力区分 |
| ポリシー | `tenant_config` 上の振る舞い定義（属性→効果） |

---

## 2. 権限レイヤー構造

```
認証/テナント解決 → データ境界(tenant_id, workspace) → 取得制御(scope) → 出力制御(回答モード)
```

- 入口（`Depends`）でテナントを確定し、以降のすべてのデータアクセスを `tenant_id` でスコープ。

---

## 3. データ境界（テナント分離）

課題要件「顧客・ワークスペース・ドキュメント・回答・引用の境界」をデータモデルで担保する。

| 境界 | 規則 | 参照カラム |
|---|---|---|
| 顧客 | 他テナントの行に到達しない | `tenant_id` |
| ワークスペース | scope内のみ検索対象 | `workspace_id` |
| ドキュメント | テナント/WSに属す | `document.tenant_id` |
| 回答 | テナントに紐づく | `answer.tenant_id` |
| 引用 | 回答が参照した資料のみ | `citation.answer_id` |

---

## 4. 属性ベースのポリシー（ABAC的）

`tenant_config` を「属性 → 効果」のポリシー集合として扱う。RBACの固定ロールでなく、**属性（顧客種別・問い合わせカテゴリ・回答モード）で振る舞いを決める**。

| 属性 | 効果（例） |
|---|---|
| `answer.citation = required` | 引用なしの断定を抑止 |
| `category = contract` | `require_human_check`（担当部署確認の導線を出す） |
| `mode = external` | 社内専用情報を出さない |
| `warnings.stale_sources = true` | 古資料参照時に警告 |

---

## 5. 出力制御（回答モード）

| モード | 内容 |
|---|---|
| external（社外向け） | 丁寧表現・社内専用情報を含めない |
| internal（社内確認用） | 根拠資料・判断理由を詳しく表示 |

- A社：社外/社内。B社：現場（短く）/管理者（詳しく）。C社：患者向け/社内確認用。**同じ仕組みの値違い**。

---

## 6. 代表的ルール

### 6-1. 引用必須（A社 B-1）
`citation: required` のとき、根拠が引けない質問は断定せず確認導線を出す。

### 6-2. カテゴリ別の確認要否（A社 B-7）
`category_policies.contract.require_human_check = true` → 「担当部署への確認が必要」を明示。

### 6-3. 高リスク制御（C社想定 / bespoke）
医療判断に踏み込む質問は回答制限し、人間確認・エスカレーション先を提示。`policies/registry.py` に名前付き登録、C社configのみ選択。

### 6-4. PII注意（C社想定 / 共通化候補）
個人情報らしき入力に注意喚起。共通部品として戻せる見込み。

---

## 7. ログ設計

### 7-1. 評価ログ

| フィールド | 内容 |
|---|---|
| tenant_id / query | どのテナントの何の質問か |
| status | needs_review / answered / no_data |
| warnings | stale / contradiction 等 |
| citations | 参照資料・更新日 |

### 7-2. 改善候補ログ

- 悪い評価・回答できなかった質問を蓄積（B/C社にも流用）。秘密情報の全文出力はしない。

---

## 8. APIレイヤー統合

- 認可・スコープは `Depends` と repository 層で一元化し、各エンドポイントに散らさない。

---

## 9. フロントエンド制御

| パターン | 説明 |
|---|---|
| モード切替の可否 | `tenant_config.answer.modes` に応じて表示 |
| 警告/ステータス表示 | config と回答メタに応じて条件付きレンダリング |
| 社内専用UI | internal モードのみ判断理由・根拠を詳細表示 |
