# 📋 機能一覧

機能数を増やすことより、**1つの顧客課題（A社）に対して設計・実装・共通化判断が明確であること**を優先する。

## 0. 前提定義

| 項目 | 内容 |
|---|---|
| プロダクト | 企業向けRAGチャット（config駆動マルチテナント） |
| 実装対象 | 標準RAG + **A社（東雲ビジネスサポート）** |
| 共通の単位 | `/chat`・パイプライン・部品は全テナント共通 |
| 差分の単位 | `tenant_config`（データ） |

---

## 1. 機能カテゴリ設計

> 各機能は ID / 機能名 / 概要 / ユーザー価値 / 優先度 / 備考（分類）で管理。
> 分類 = 設定吸収 / 共通部品 / bespoke のどれか（`multi-tenant-design.md`）。

### 【カテゴリA：標準RAG共通機能（必須）】

| ID | 機能名 | 概要 | ユーザー価値 | 優先度 | 分類 |
|---|---|---|---|---|---|
| A-1 | ナレッジ取り込み | doc/FAQ/メモを取り込む | 社内資料を回答に使える | P0 | 共通 |
| A-2 | 分割・索引・検索 | 文書分割→埋め込み→近傍検索 | 関連資料を引ける | P0 | 共通 |
| A-3 | 質問応答 | 質問に回答を返す | 問い合わせ対応 | P0 | 共通 |
| A-4 | 引用表示 | 回答に根拠・引用を表示 | 根拠を確認できる | P0 | 共通 |
| A-5 | 非断定 | 根拠不足時に断定しない | 誤情報を防ぐ | P0 | 共通 |
| A-6 | チャット画面 | 簡単なチャットUI | 利用できる | P0 | 共通 |
| A-7 | 境界設計 | 顧客/WS/doc/回答/引用の境界 | 分離・再利用の土台 | P0 | 共通 |
| A-8 | ローカル動作 | mock providerで動作確認 | APIキー無しで確認 | P0 | 共通 |

### 【カテゴリB：A社向け追加機能（今回実装）】

| ID | 機能名 | 概要 | ユーザー価値 | 優先度 | 分類 |
|---|---|---|---|---|---|
| B-1 | 引用必須化 | 回答に必ず根拠を表示 | 担当者が根拠で迷わない | P0 | 設定吸収 |
| B-2 | 根拠不足ステータス | 弱い根拠は「確認が必要」 | 断定を防ぐ | P0 | 共通部品(ground_check) |
| B-3 | 古資料警告 | 古い/更新待ち資料に警告 | 古い規程を誤用しない | P0 | 共通部品(stale_warning) |
| B-4 | 資料矛盾表示 | 複数資料の矛盾可能性を表示 | 矛盾に気づける | P1 | 共通部品(contradiction_check) |
| B-5 | フィードバック保存 | 良い/悪い + 理由カテゴリ+コメント | 改善対象が見える | P1 | 共通 |
| B-6 | 改善候補一覧 | 改善が必要な回答を一覧化 | 品質改善の運用 | P1 | 共通 |
| B-7 | カテゴリ別方針 | 請求/契約/規程ごとに方針 | カテゴリ最適な回答 | P1 | 設定吸収 |
| B-8 | 回答モード | 社外向け/社内確認用を分ける | 用途別の回答 | P1 | 設定吸収 |
| B-9 | 資料メタ表示 | 資料名・更新日・該当箇所 | 出所を確認できる | P1 | 設定吸収 |

### 【カテゴリC：共通プロダクト機能として戻すもの】

B社・C社でも再利用できるため、標準RAG側の共通機能（部品 or 設定項目）に昇格する。

| ID | 機能名 | 戻し方 | 優先度 | 備考 |
|---|---|---|---|---|
| C-1 | 引用必須 | `tenant_config.answer.citation` | P0 | 全社が設定で有効化可 |
| C-2 | 根拠不足ステータス | 共通step `ground_check` | P0 | |
| C-3 | 資料警告 | 共通step `stale_warning` | P0 | |
| C-4 | 回答モード | `tenant_config.answer.modes` | P1 | B社=現場/管理者 等に流用 |
| C-5 | フィードバック/改善候補 | 共通機能 | P1 | B/C社も「回答できなかった質問」蓄積に流用 |

### 【カテゴリD：今回作らない / Nice to have】

| ID | 機能名 | 概要 | 優先度 | 分類 |
|---|---|---|---|---|
| D-1 | B社：部署/設備別スコープ | `search.scope` で表現 | P2 | 設定吸収（実装は次回） |
| D-2 | B社：危険作業の検知警告 | bespoke部品候補 | P3 | 顧客固有 |
| D-3 | C社：医療判断の回答制限 | bespoke部品候補 | P3 | 顧客固有 |

## 2. 優先順位の定義基準

| 優先度 | 意味 | 判断基準 |
|---|---|---|
| P0 | 必須（MVP） | 標準RAG成立 / A社の中核要望 |
| P1 | 重要 | A社の価値を実演に足る |
| P2 | あると良い | 設定で表現できると示せる |
| P3 | Nice to have | 今回は作らず判断メモで言及 |

---

## 3. フェーズ整理

| フェーズ | 含む機能 | ゴール |
|---|---|---|
| Phase 0 | A-1〜A-8 | 標準RAGが成立 |
| Phase 1 | B-1, B-2, B-3, B-9 | A社の中核（引用必須・非断定・警告・メタ） |
| Phase 2 | B-4〜B-8 | フィードバック/モード/カテゴリ方針 |
| Phase 3 | C-1〜C-5 | 共通へ還元・設定化を整理 |
